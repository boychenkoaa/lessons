## Основные требования
- Роботы независимы и могут находиться в одной и той же зоне. Роботов не нужно никаким образом синхронизировать между собой.
- Последовательность действий одного робота должна сохраняться. Параллелизм и перемешивание допускаются только для разных роботов.

## Основные архитектурные проблемы

1. Разделение ответственности между инфраструктурой и бизнес-логикой. 

- С одной стороны, для принятия доменного решения процессору нужно знать текущее состояние, с другой, давайть доступ к истории для каждого процессора. 
- Компромисс -- запись в хранилище как сайд-эффект для домена, и  кеширование последнего состояния для каждого робота (тоже делалось отдельным инфра-процессором). Отсюда же идут отдельные инфраструктурные процессоры, шина событий и отдельная обрабтка их ошибок (потоки после инфраструктурного сбоя пропускают события "вхолостую").

2. Многопоточность. 
- Организация критических секций (все, к чему потоки имеют совместный доступ)
- Штатное завершение работы пула потоков (рассылка сентинелов как признаков конца последовательности событий, и их обработка потоками) -- `Dispatcher.shutdown`   [event_service.py](event_service.py)
- Организация шардинга для последовательной обработки событий одного робота -- `Dispatcher._run` 
- Организация атомарности цикла запрос -> результат -> применение. Чтобы в зазор между добавлением в очередь результата А и его применением не влез запрос B со своим применением.
