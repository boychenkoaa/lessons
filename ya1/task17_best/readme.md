# АТД

Сначала подумал, что от меня нужен лексический анализатор, но потом подумал что задача все же проще :)

Общее замечание:
- Нехорошее" с точки зрения ООП место -- это явная зависимость DSL от конкретных реализаций `Move` и `RobotState`. + эти классы мутабельны.
- Делаем слой - фабрику команд, функции типа make_turn, make_move... Полноценную  необязательно, функций вполне достаточно. Можно при желании подменить Move на DoubleMove, например. Инъектировать их в DSL пока не буду, хотя по хорошему -- нужно.
- Делаем также команды `_Move`, `_Turn`, .. и состояние робота `RobotState` полноценными классами с приватными полями. Через это будет псевдо-иммутабельность. Они приватны -- видны только в модуле `robot_commands`
- Публичными у команд оставляем только метод `interpret()` (у абстрактной команды Command он абстрактный)
- Команды делим на терминальную (`_Stop`) и команды робота `RobotCmd`. `RobotCmd` тоже АТД с переопределяемым методом `_do()`.
