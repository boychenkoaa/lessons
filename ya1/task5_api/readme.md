# Задание 5 - "хипстерский" API

Надеюсь, я правильно понял задачу.

### Общие соображения
- Задача клиенского API -- реализовывать высокоуровневый и типовой для клиентов функционал. Но при этом с возможностью его доработки интеграторами / квалифицированными пользователями.
- Естественным шагом кажется не вызывать функции движения робота напрямую (даже если это технически возможно), а генерировать для робота текстовые программы. Или имелось в виду, что хипстеры именно так и делают?
- Во первых, синтаксис языка взаимодействия с роботом гораздо более консервативен, чем конкретная его реализация. 
- Во вторых, мы уходим от конкретики реализации транспортного уровня. Когда передается plain text -- нам становится безразлично, будут ли это прямые вызовы, RPC, REST или потоки ввода-вывода. Все они годятся для передачи текста :)
- Например, нашим клиентам нужно чистить прямоугольные комнаты. Тогда функциия `clean_rect`, будет вызывать исполнение программы `clean_rect_program`. Последовательность перемещений и поворотов внутри нее соответствует набору вложенных прямоугольников (снаружи внутрь, от большего к меньшему). Прямоугольники проходится сначала щеткой, потом шампунем, потом водой.
- Еще нужны базовые функции работы с сессией `connect` / `disconnect` и дата-класс `Session`, хранящий состояние робота и флаг, открыто ли соединение.
- Для продвинутых юзеров оставим запас на кастомизацию -- функцию `run_program`.


### Особенности реализации
Сделал два слоя -- два модуля.
- Первый слой [programs.py](programs.py) -- функции-генераторы типовых программ.
- Второй слой [robot_api.py](robot_api.py). -- сами клиентские функции API.
- [pure_robot.py](pure_robot.py) Немного изменил пробрасываемые в transfer сообщения, чтобы ограничить кол-во знаков у вещественных чисел. Также добавил условие, чтобы угол всегда был от 0 до 360 - при повороте вычисляем значение по модулю.
